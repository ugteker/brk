## 1. Requirements (cumulative, append-only unless explicitly superseded)

### Product Direction
- ChatTrader bots are **AI agents**: they crawl configured sources, feed evidence + a
  system prompt into the **Anthropic Claude API (Sonnet model)**, and produce a
  structured report of stock signals.
- Report output = **long/short signal per symbol**, with **confidence**,
  **rationale**, and **source citations/timecodes**. Informational only —
  **no automated trade execution, brokerage write-back, or live market-data platform.**
- If a bot crawls a podcast, the report must reflect everything discussed and
  indicate which symbols are suggested long vs. short based on the conversation.
- User has an Anthropic Claude API key (API-based account) — integration must call
  the real Anthropic API in production, but stay **testable without a live Claude
  dependency** (inject a fake client in tests).

### Architecture / Backend
- Stack: TypeScript, Fastify, Prisma, **SQLite** for local dev (not Postgres).
- Keep the existing scheduler/worker shape — no separate orchestration stack.
- Existing Prisma model names (`Bot`, `BotRun`, etc.) may remain as-is during the
  transition even as product language shifts from "bot" to "agent".
- The web app must keep proxying `/api` requests to the API during local dev.
- The dashboard must load persisted agents/reports (not stay local-only).

### Frontend
- Primary UI library: **Ant Design** (chosen as "most popular" per user request for
  something "fancy"), replacing the shadcn-style component set for the redesigned
  screens.
- Agent creation/editing is split by domain ownership:
  - **Agents** hub: character/persona/prompt identity
  - **Sources** hub: source library and source-level config
  - **Playbooks** hub: source selection + schedule/recipients + execution policy
- **Reports browsing view**: once a bot/agent produces a report, users must be able
  to navigate through all related reports for that agent. Each report list item
  must show:
  - Badges for every stock symbol mentioned in that report (long/short colored).
  - The date the report was created.
  - A very short summary/title/headline describing the report.
  - A visual indicator of how confident/reliable the report's signals are
    (e.g. a progress/gauge-style confidence indicator), not just a number.
- **Bot pause/resume**: each bot must have a pause/resume icon control (not a
  text button) so the user can toggle a bot between active and disabled
  directly from the dashboard/detail view.
- **Wizard step headers**: the 6-step agent setup wizard's step headers must
  use icons + short titles (not long squeezed text labels).
- **Dark theme correctness**: dark mode must apply consistently to *all*
  elements, including Ant Design inputs/cards/selects — not just
  Tailwind-styled elements.
- **Theme toggle control**: the theme switch must be a single icon button
  (sun/moon), not a dropdown/select.
- **Trading persona prompt library**: the wizard's System prompt step must
  offer multiple full, ready-to-use system prompts ("personas") with
  different trading characters/risk profiles that the user can pick from,
  each of which also sets a sensible default risk level.
- **Full-field editing across hubs**: editing must keep ownership boundaries:
  agent identity/persona in Agents, source details in Sources, and
  schedule/recipients/execution in Playbooks.
- **Smart, AI-assisted crawling**: agents must not reprocess the same content
  every run. Feed-like sources (RSS/Atom, podcast feeds) use a deterministic
  cursor (seen-item-ID tracking); non-feed sources (blogs/listing pages) get
  a one-time Claude "site inspection" call that produces a persisted crawl
  config (selectors/patterns), applied deterministically on every subsequent
  run, with self-healing re-inspection if the site's markup changes (capped
  at 1 reinspection attempt per source per 24h).
- **Fail-fast source probing**: the wizard must let the user "Test source" a
  URL/feed *before* saving the agent, showing whether it's reachable, what
  kind of source it looks like (feed/listing page/single page/unknown), an
  item count, and a confidence score — without waiting for the first
  scheduled run.
- **Per-source episode/item counter**: each source must have a configurable
  "episodes/items per run" cap (`maxItems`, default 1, range 1-10) instead of
  a hardcoded constant, surfaced directly in the wizard next to the source
  URL.
- **Manual run-now**: users must be able to trigger an immediate run from the
  **Playbooks** hub, not just wait for the next scheduled tick.
- **Agent change confirmation emails**: recipients must get an email whenever
  an agent they're subscribed to is created or updated.
- **Authentication**: username/password signup+login with two-step email
  confirmation, Google social login, forgot/reset password, and an
  admin-only user management view (lock/unlock/delete users) restricted to
  the `ADMIN_EMAIL` configured in env. All `/api/agents`/`/api/admin` routes
  must be gated behind a valid session.
- **Runs/Reports ownership**: Runs and Reports belong to the **Playbooks**
  hub (not Agents). The Playbooks view must show run history (status,
  scheduled/started/finished time, duration, retry count), real failure
  reasons, crawled-content previews with download, and links to related
  reports.
- **Weekly schedule mode**: in addition to interval and daily-time modes, the
  wizard must support a weekly mode where the user picks specific days of
  the week plus a daily time.
- **Multi-recipient emails**: the wizard's recipients field must be a proper
  multi-email tags input (not a single plain-text field the user has to know
  to comma-separate), and must default to the logged-in user's own email for
  new agents.
- **Active toggle must take effect**: the wizard's "Active" toggle must
  actually set the agent's status (`active`/`disabled`) on create/update —
  not be purely cosmetic — while the per-source "Enabled" toggle and the
  "Active" toggle are both intentionally kept (user explicitly reversed
  earlier removal decisions for both).
- **Live-updating Playbooks outputs**: Playbooks-owned Runs/Reports views
  should reflect new reports / run-status changes without requiring a manual
  page reload.
- **Robust JSON parsing of Claude responses**: the analysis pipeline must
  tolerate Claude wrapping its JSON response in markdown code fences
  (` ```json ... ``` `) instead of crashing with a raw `JSON.parse` error.
- **YouTube as a source type**: alongside `web_urls` and `podcast_feeds`,
  agents must support YouTube sources — a single video, or a channel/playlist
  URL that's auto-resolved to list its videos so the N most recent
  *unseen* episodes (per the existing episode-count setting) are crawled
  automatically, with full video transcripts fetched (not just show notes),
  rather than requiring the user to add every video URL manually.

### Process / Workflow (user-directed, applies across this session and future work)
- **All requirements must live in a single markdown source of truth (this file)** —
  do not scatter requirements across chat history only.
- **This file must be updated automatically as progress is made** (after each task,
  not just at the end).
- Since this repo has no git history, implementation proceeds via direct
  session-based execution (TDD per task) rather than full subagent-driven-development
  tooling, which assumes a git repo.

## 5. Known Issues / Outstanding Follow-ups
- **YouTube auto-generated ("asr") captions return an empty transcript in production, and this
  appears to be an IP/anti-bot block rather than a fixable code bug.** The
  `fetchYouTubeTranscript()` flow correctly locates the video's caption track and its signed
  `/api/timedtext` URL from the watch page HTML, but fetching that URL returns HTTP 200 with a
  **0-byte body** for auto-generated caption tracks on real videos — reproduced live via curl/node
  against multiple real YouTube videos, with and without a browser User-Agent, Referer/Origin
  headers, and session cookies from the watch-page request; none of these made a difference.
  **Follow-up investigation (this session): tried the modern InnerTube `youtubei/v1/get_transcript`
  API as a replacement** — both a hand-built protobuf `params` field and the *exact* `params` value
  YouTube itself embeds in the watch page's `getTranscriptEndpoint` JSON (extracted live from a
  real video, using the real `INNERTUBE_API_KEY`/`INNERTUBE_CONTEXT_CLIENT_VERSION` from the same
  page) were POSTed to `youtubei/v1/get_transcript`. **Both attempts failed identically** with
  `400 FAILED_PRECONDITION` ("Precondition check failed"), even though the params/key were sourced
  directly from a fresh, valid watch-page load. Since the *literal* server-supplied params still
  gets rejected, this isn't a params-encoding bug on our side — it strongly indicates YouTube is
  rejecting the request based on request **context/origin** (e.g. missing a browser-session-bound
  visitor/session token, TLS/JA3 fingerprinting, or — most likely, matching widely-reported behavior
  of libraries like `youtube-transcript-api` — an IP-reputation block against cloud/datacenter IP
  ranges, which this server runs on). **Effect**: agent runs against YouTube sources with only
  auto-generated captions will report "no new content" every run, because every candidate video's
  transcript fetch fails, gets logged as a warning, and the video is still marked "seen" (so it's
  not retried forever) — this can look like the crawler is silently exhausting a channel's recent
  videos without ever producing evidence. **Not fixable via request tweaks alone from this
  environment.** Realistic paths forward, none implemented yet:
  1. Route transcript fetches through a residential/rotating proxy (adds cost/complexity).
  2. Use a paid third-party YouTube-transcript API service.
  3. Fall back to crawling the show-notes/description page content only for YouTube sources (already
     the behavior when transcript fetch fails), and set user expectations that full YouTube
     transcripts are not currently retrievable from this server's IP.
  4. Re-test from a non-datacenter IP (e.g. a residential network) to confirm the IP-block theory
     before investing further.
